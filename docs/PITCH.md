# Pitch

## 30 秒介绍版本

GitDian 是一个把 GitHub 开源仓库转成“创业机会工作流”的本地分析台。它不只是看仓库，而是把仓库采集、粗筛、AI 分析、任务追踪和收藏沉淀串成一个闭环，让我们能更快判断哪些项目值得继续研究、重做或商业化包装。

## 2 分钟版本

GitHub 上有大量公开项目，但真正困难的不是“找到仓库”，而是“从海量仓库里快速挑出值得继续看的机会”。GitDian 解决的是这个问题。

它先做 GitHub 仓库采集，再用 Fast Filter 做第一层粗筛，然后用本地大模型输出三类更接近产品判断的信息：完整性、机会匹配度，以及这个仓库能抽象成什么产品点子。整个过程不是同步卡页面，而是通过异步任务系统运行，所以可以看到任务状态、失败原因、重试和取消。

从演示角度，它已经不是一个概念原型，而是一个能真实启动、真实跑任务、真实看到结果的本地工作台。现在它适合做内部演示、技术展示、方案讨论，也适合作为下一阶段继续迭代的底座。

## 5 分钟演示讲解脚本

### 开场

今天想演示的不是一个普通的 GitHub 浏览器，而是一个“从开源仓库发现创业机会”的工作台。核心价值是把原来很碎、很靠人工经验的判断过程，收敛成一条可以重复执行的流程。

### 第一步：首页

先看首页。这里不是单纯列仓库，而是把仓库候选、工作流入口和最近任务放在一起。你可以把它理解成一个开源项目发现仪表台，目标不是阅读，而是筛选。

### 第二步：GitHub 一键采集

接下来我触发一次 GitHub 采集。这个动作现在已经是异步任务了，所以页面不会一直卡着等。系统会先创建 Job，再交给 Worker 跑，适合做批量数据处理，也更适合真实演示。

### 第三步：任务页

然后切到 `/jobs`。这里能看到任务状态、任务详情、错误信息、重试和取消。对演示来说这一页很重要，因为它把“后台到底有没有在工作”这件事透明化了。

### 第四步：仓库详情页

再打开一个仓库详情页。这里能看到 README、目录摘要、粗筛结果和已经落库的分析结果。也就是说，我们不是只生成一次文本，而是把结果和仓库对象绑定起来，方便后续持续跟进。

### 第五步：一键分析

我在详情页触发一次完整分析。这里会跑 Fast Filter、Completeness、Idea Fit 和 Idea Extract。Fast Filter 更像第一层判断，后面三步更接近“这个项目能不能拿来做产品机会”的分析。

### 第六步：收藏与工作流

如果某个仓库值得继续看，就可以收藏、设置优先级、写备注。这样仓库就从“看过”变成“可以继续推进的机会项”。

### 第七步：设置页

最后看设置页。这里能看到数据库、GitHub 和本地 oMLX 的健康状态。这个点很关键，因为它说明这个项目不是只在 PPT 上能讲，而是真正在本机跑通了完整链路。

### 收尾

所以如果一句话总结，GitDian 做的事情是：把 GitHub 的海量开源仓库，变成一个可以筛选、分析、追踪和沉淀的创业机会工作流。现在这个版本已经适合做内部试运行、技术展示和下一阶段产品化迭代。
